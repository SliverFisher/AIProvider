# 情报雷达设计

日期：2026-07-21

## 目标

在 AIProvider 左侧一级导航的“运营与工具”分组中增加“情报雷达”，将 GitHub、RSS、普通网页、YouTube、B站、小红书、X、Reddit 及 RSSHub 长尾来源统一为可订阅、可检索、可收藏的信息流。

首个完整版本必须使用真实来源、真实账号状态和真实采集结果。任何渠道不可用时应明确失败，禁止生成演示数据、静默跳过、切换备用实现或以其他来源结果代替。

## 开源复用边界

- 将 MIT 许可证的 Agent-Reach 作为采集能力参考和可复用代码来源，保留原始 `LICENSE`、版权声明和来源说明。
- 复用代码放在 `third_party/agent-reach`，项目适配代码放在独立的情报采集模块中，避免直接修改上游文件。
- 每个平台只启用一个明确的上游实现，禁用 Agent-Reach 的自动备用后端选择。
- RSSHub 和 Karakeep 的 AGPL-3.0 代码不复制进 AIProvider。RSSHub 只能作为独立部署的外部服务通过 HTTP/RSS 协议调用；Karakeep 只用于产品交互参考。
- 每个实际引入的上游组件在实施前记录仓库地址、固定版本或提交哈希、许可证及项目内用途。

## 用户界面

### 导航

左侧一级导航新增“情报雷达”，归入“运营与工具”，路由为 `/intelligence`。移动端导航必须保持可达和可滚动。

### 页面分区

页面包含四个明确分区：

1. **今日雷达**：统一时间流，支持平台、订阅、时间、标签和已读状态筛选。
2. **订阅管理**：创建、修改、启停和删除关键词、账号、频道、仓库及 RSS 地址订阅。
3. **我的收藏**：管理收藏、标签、备注、已读和稍后阅读状态。
4. **渠道状态**：展示平台实现、账号状态、最近成功时间、最近错误和下一次计划时间。

所有搜索输入必须复用 `AIProvider-front/src/UiSearchField.jsx`。实时筛选不增加提交按钮。页面必须消费 `uiTheme` 语义变量，并使用原生可交互元素、可见焦点和准确无障碍名称。

## 系统架构

### 统一后端

Spring 后端负责：

- 订阅和账号配置；
- Cookie、Token、StorageState 和代理配置的加密存储；
- 定时任务生成与调度；
- 采集器调用；
- 统一数据转换、批量去重、批量写入；
- 搜索、分页、收藏、标签和已读状态；
- 渠道健康和任务历史；
- 可选摘要任务。

PC 和手机端只访问同一个后端，因此共享订阅、账号登录状态和采集结果。

### 采集执行

- GitHub：固定使用 GitHub 官方 `gh` CLI。
- RSS/Atom：固定使用 Agent-Reach 当前采用的 feedparser 路径。
- 普通网页：固定使用 Agent-Reach 当前采用的 Jina Reader 路径。
- YouTube：固定使用 yt-dlp 获取元数据和可用字幕。
- B站：固定使用 bili-cli；不得在失败时切换 yt-dlp。
- 小红书：固定使用适合服务器登录态的 xiaohongshu-mcp。
- X：固定使用 twitter-cli，不并行保留现有 Web 适配器作为情报雷达采集路径。
- Reddit：固定使用 rdt-cli 和后端保存的登录态。
- 微信、微博、知乎等长尾来源：通过独立 RSSHub 实例输出标准 RSS，再由统一 RSS 采集器读取。

实施前必须通过本机和 AWS 可用性探测验证上述固定实现。探测失败时停止对应渠道的实施并报告阻塞，不选择备用实现；正式运行时同样不自动切换。

### Bridge 边界

Bridge 只在某个平台必须使用本机浏览器完成登录时参与登录态采集。登录成功后，凭据通过后端授权接口上传并加密保存。Bridge 不长期保存正式 Cookie，不承担常驻采集任务，也不判断后端记录是否存在。

## 凭据安全

- 复用现有 `ContentPlatformSecretCipher` 的 AES-GCM 模式和生产环境密钥管理方式。
- 数据库只保存密文、非敏感提示、状态和时间，不保存明文凭据。
- API 响应只暴露“是否已配置”、账号提示、有效状态和错误，不返回原始 Cookie、Token 或 StorageState。
- 后端仅在调用固定采集器前于内存中解密，不写入普通临时文件、命令行参数或日志。
- 如上游工具必须读取文件，实施时必须使用权限受限、任务结束立即删除的明确临时文件，并增加删除验证；不得使用共享或长期凭据文件。
- 平台凭据、代理凭据和第三方密钥均禁止进入源码、测试快照、构建产物和 Git。
- 账号更新采用“留空保持原凭据，提供新值才替换”的语义。

## 数据模型

所有实体使用后端数据库分配的稳定数值 ID。

### IntelligenceCredentials

保存平台、显示名称、适配器类型、加密凭据、凭据提示、启用状态、连接状态、最近验证时间和最近错误。

### IntelligenceSubscriptions

保存平台、凭据 ID、订阅类型、外部标识、查询条件、名称、采集间隔、单次上限、启用状态和最近采集状态。

### IntelligenceItems

保存订阅 ID、平台、平台内容 ID、原链接、标题、摘要、作者、发布时间、采集时间、标准化元数据和内容指纹。数据库对“平台 + 平台内容 ID”建立唯一约束；无稳定平台 ID 时使用标准化原链接的唯一约束。

### IntelligenceCollectionRuns

保存任务类型、订阅 ID、状态、开始与结束时间、请求数量、获取数量、去重数量、实际写入数量、错误代码和截断后的非敏感错误信息。

### IntelligenceBookmarks

保存条目 ID、收藏状态、已读状态、稍后阅读状态、备注和更新时间。一个条目只有一条用户状态记录。

### IntelligenceTags 与 IntelligenceItemTags

保存标签及条目标签关系。批量加标签、移除标签和删除条目状态必须一次提交 ID 列表并使用批量 SQL。

### IntelligenceChannelStates

保存平台唯一状态、固定实现、运行位置、认证状态、最近检查时间、最近成功时间和最近错误。

## 数据流

1. 调度器按启用的订阅创建独立采集运行记录。
2. 服务根据订阅平台调用唯一的固定采集器。
3. 采集器返回统一条目列表和来源元数据，不直接写数据库。
4. 后端校验字段、标准化 URL 和外部 ID，并在一次数据库事务中批量去重和批量写入。
5. 后端记录请求数量、获取数量、去重数量和实际写入数量。
6. 实际写入数量与合法待写入数量不一致时，任务标记失败并记录警告日志，禁止返回成功。
7. 前端通过分页 API 获取条目；查询、平台、订阅、标签、时间和已读条件均在后端筛选。

## API 边界

API 使用 `/api/intelligence` 前缀，至少包含：

- 分页查询条目；
- 创建、修改、启停和删除订阅；
- 创建、更新、验证和停用凭据；
- 手动触发单个或批量订阅采集；
- 查询采集运行历史；
- 批量更新收藏、已读、稍后阅读和标签；
- 查询渠道状态。

任何批量修改都接收去重后的数值 `ids` 列表，后端使用批量 SQL，并校验实际影响行数。凭据查询 API 不提供密文或明文返回字段。

## 错误处理与可观测性

- 渠道错误使用稳定错误代码区分：缺少凭据、凭据过期、限流、IP 拦截、代理错误、上游不可用、解析失败和数据校验失败。
- 一个渠道失败不修改其他渠道的运行状态。
- 页面保留旧数据，但必须显示该渠道当前采集失败和失败时间，不能把旧数据标为最新成功结果。
- 所有具体业务操作记录结构化日志，至少包含操作名称、渠道、凭据 ID、订阅 ID、运行 ID、请求数量、获取数量、去重数量和实际影响行数。
- 日志禁止记录密钥、Cookie、Token、StorageState、完整正文、完整 Prompt、文件内容及认证请求头。

## 测试与验收

### 后端

- 凭据加密、更新不回显和解密失败测试；
- 每个固定采集器的契约测试；
- 批量去重、唯一约束和影响行数不一致测试；
- 调度、状态转换和独立渠道失败测试；
- 结构化日志字段存在且敏感值缺失测试；
- API 分页和组合筛选测试。

### 前端

- 四个分区的加载、空状态、错误状态和分页测试；
- 实时搜索和组合筛选测试；
- 订阅、凭据和批量状态操作测试；
- 渠道失败时保留旧数据并显示失败状态测试；
- 更新并通过 `uiGate.test.js`，覆盖搜索复用、主题、导航、移动端、响应式和交互语义。

### 真实验收

每个平台必须分别证明：真实账号或公开来源可访问、创建订阅、执行采集、数据库产生真实条目、页面展示原链接与发布时间、再次采集正确去重、渠道失败时显示明确错误。构建和单元测试不能代替真实来源验收。

## 实施阶段

1. 固定第三方版本和许可证清单，探测并选定每个平台唯一实现。
2. 建立数据库、统一 DTO、批量持久化、凭据库和任务状态机。
3. 接入 GitHub、RSS、普通网页和 YouTube。
4. 完成情报雷达四分区页面。
5. 接入 B站、小红书、X 和 Reddit。
6. 独立部署并接入 RSSHub 长尾来源。
7. 完成收藏、标签、摘要、调度和渠道状态。
8. 运行本次改动覆盖的本地测试、UI 门禁和逐渠道真实验收。

生产发布不属于本设计的默认范围，只有用户明确要求发布后才进入版本递增、提交、推送和生产部署流程。
