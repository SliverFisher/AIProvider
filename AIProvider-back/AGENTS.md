# Java 后端开发规范

## 总原则

这是一个需要长期维护的正式项目，不是 Demo。

开发前必须先阅读现有代码，找到至少 2 个结构完整的现有模块作为参考。
新代码必须遵循现有项目的分层、命名、返回结构、异常处理和数据库规范。
除非明确要求，不得为了快速完成功能而简化项目架构。

## 分层要求

### Controller

Controller 必须保持轻量，只允许：

- 接收和校验请求参数
- 获取当前登录用户信息
- 调用 Service
- 返回统一响应对象

Controller 禁止：

- 编写业务判断和业务流程
- 直接调用 Mapper 或 Repository
- 直接操作数据库
- 编写事务
- 组装复杂业务数据
- 编写大量循环、条件分支或数据转换
- 直接返回数据库 Entity

### Service

Service 负责：

- 业务规则
- 业务流程编排
- 权限和状态判断
- 调用 Mapper
- 事务控制
- DTO、Entity、VO 之间的转换协调
- 跨模块业务处理

需要事务时，在 Service 层使用 `@Transactional`。

### Mapper / Repository

Mapper 只负责数据库访问。

禁止在 Mapper 中编写业务决策、权限判断或业务流程。

### 数据对象

严格区分：

- DTO：接收请求参数
- VO：返回给前端
- Entity / DO：数据库实体
- Query：复杂查询条件

禁止直接把 Entity 暴露给前端。

## 开发要求

- 优先复用现有组件、工具类和统一返回结构
- 不得重复创建已有功能
- 不得随意引入新的框架或架构
- 不得因为功能简单就把所有代码放进 Controller
- 不得未经说明大规模修改现有目录结构
- 数据库修改必须提供迁移脚本
- 敏感配置必须使用环境变量
- 保持方法职责单一，避免超长方法
- 重要业务逻辑必须有注释
- 异常必须使用项目已有的统一异常处理方式

## 工作流程

每次开发前：

1. 阅读 AGENTS.md
2. 检查现有同类模块
3. 说明准备修改哪些文件
4. 说明 Controller、Service、Mapper 各自承担什么职责
5. 再开始编码

完成后：

1. 检查 Controller 是否包含业务逻辑
2. 检查 Controller 是否直接调用 Mapper
3. 检查是否直接返回 Entity
4. 运行后端构建和测试
5. 汇报修改内容和验证结果

发现现有代码不符合以上规范时，不要继续复制错误写法，应先说明问题并按规范重构。
